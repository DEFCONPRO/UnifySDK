◆ emberAfDeviceTableNewDeviceJoinHandler()

void emberAfDeviceTableNewDeviceJoinHandler ( EmberNodeId  newNodeId,
EmberEUI64  newNodeEui64 
)

Informs the device table that a new device joined.

When a device joins, rejoins, or sends an end-device announce, call this function in the device table. If the device is new, it will generate a new device table entry and kick off the discovery process. If the device is not new, the device-table will make sure the node ID matches the EUI64 and make an update if necessary. It will also check to verify the discovery process has completed and attempt to complete it.

Parameters
newNodeIdThe node ID of the newly joined device.
newNodeEui64The EUI64 of the newly joined device.

Informs the device table that a new device joined.

  1. a new device has joined.
  2. device nodeID has changed.
Parameters
newNodeId
newNodeEui64

◆ emberAfPluginDeviceTableInitiateRouteRepair()

void emberAfPluginDeviceTableInitiateRouteRepair ( EmberNodeId  nodeId)

Kicks off device table route repair.

A message was sent to the device and failed. Therefore, the route repair will be attempted and the node marked as unresponsive. Currently, route repair simply sends a broadcast message looking for the node ID based on the EUI64. It may be replaced with something more sophisticated in the future.

Parameters
nodeIdThe node ID of the newly unresponsive node.

◆ emberAfPluginDeviceTableMessageReceived()

void emberAfPluginDeviceTableMessageReceived ( EmberNodeId  nodeId)

Informs the device table that a message was received.

The device table keeps track of whether a device has become unresponsive. Hearing from a known device will cause the device table to transition the device state from "unresponsive" to "joined". Hearing from an unknown device will cause the device table to discover the device, potentially fixing routing issues (i.e., if the node ID has changed).

Parameters
nodeIdThe node ID from which the last incoming message came.
